A10:2021 — Server-Side Request Forgery (SSRF)

Not Solved
Goal (informational level): a "profile picture from URL" feature fetches whatever URL you give it, server-side, with no restriction on the destination. Point it at an internal-only address instead of a real image, and the server will happily fetch it for you.

Import Profile Picture from URL

show hint
The server has no allow-list, so it will fetch internal/private addresses just like a public one. Try one of:
http://127.0.0.1/admin · http://localhost:8080/ · http://169.254.169.254/latest/meta-data/ (cloud metadata endpoint) · file:///etc/passwd 

        
🎉 The server fetched an internal resource on your behalf. In a real environment this can expose internal admin panels, cloud credentials, or local files — because the app trusted a user-supplied URL as if it were safe. That's SSRF.


        Mitigations: allow-list external destinations only, block requests to private/link-local IP ranges and the file:///gopher:// schemes, and never let user input decide what the server connects to.
